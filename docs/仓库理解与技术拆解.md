# 仓库理解与技术拆解（第一性原理）

## 1. 这个仓库在解决什么最小问题

从第一性原理看，这个项目不是在“做自动化运维”，而是在先回答一个更基础的问题：

> **在 Windows 主机上，如何稳定、低侵入地判断某个目标云桌面客户端是否处于“可用桌面状态”？**

拆成最小可验证子问题只有三个：

1. 目标进程是否存在（Process）。
2. 目标窗口是否可定位（Window）。
3. 该窗口是否满足“可用桌面”的最小规则（State Classification）。

当前仓库就是围绕这三个问题构建 MVP，优先“可观测、可解释、可迭代”，而不是一次性做复杂能力。

---

## 2. 当前实现边界（Scope）

### 已包含

- `GuardService`：.NET 8 Worker 常驻轮询服务。
- 基于进程名/可执行路径的进程筛选与候选选择。
- 基于 Win32 元数据的顶层窗口探测与打分选择。
- 会话状态四分类：
  - `NotRunning`
  - `ProcessOnly`
  - `ClientVisibleButUnknown`
  - `DesktopReady`
- 每轮输出摘要 + 结构化日志（便于后续规则收敛）。

### 明确不做（MVP 阶段）

- 不做 webhook 通知。
- 不在主流程内做 OCR 判定。
- 不做自动恢复、重启、前台激活、输入模拟。
- 不做复杂异常诊断。

---

## 3. 仓库结构与职责

- `src/GuardService`
  - 主体守护服务（生产路径）。
- `src/WindowInspector`
  - 只读 Win32 窗口树采样工具，输出 Markdown，帮助理解目标客户端 HWND 结构。
- `src/OcrProbe`
  - 实验性固定区域 OCR 工具，输出 Markdown + JSON + 截图，验证关键词/状态识别可行性。
- `docs/recognition-mvp.md`
  - MVP 规则定义、状态定义和验证清单。
- `docs/samples/`
  - 采样结果（窗口元素、OCR 样本等）。

---

## 4. 主链路数据流（GuardService）

每个轮询周期严格按以下顺序执行：

1. `ProcessMonitor.ProbeAsync`  
   输入：配置（目标进程名、可执行路径）  
   输出：`ProcessSnapshot`

2. `WindowProbe.ProbeAsync`  
   输入：`ProcessSnapshot`  
   输出：`WindowSnapshot`

3. `SessionClassifier.Classify`  
   输入：`ProcessSnapshot + WindowSnapshot`  
   输出：`SessionState`

4. `Worker` 统一打包  
   输出：`GuardCycleResult`（摘要 + JSON payload 日志）

这条链路的核心特点是：**每一步都可单独解释，不依赖黑盒模型**。

## 5. 关键判定规则（可解释）

## 5.1 进程选择规则

优先级如下：

1. 如果配置了 `TargetExecutablePath`，必须精确路径匹配。
2. 若仅有一个候选，直接选它。
3. 若多个候选且仅一个有主窗口句柄，选它。
4. 否则选“最近启动”的候选。

并且完整候选列表会保留在快照中，便于复盘“为何选中它”。

## 5.2 窗口选择规则

对选中进程的所有顶层窗口按以下顺序排序：

1. 可见优先
2. 非最小化优先
3. 面积更大优先
4. 句柄值作为确定性 tie-breaker

## 5.3 `DesktopReady` 判定规则

必须满足：

- 若 `RequireVisibleWindow=true`，窗口必须可见。
- 若 `AllowMinimizedWindow=false`，窗口不能最小化。
- 窗口大小不低于 `MinimumDesktopWidth x MinimumDesktopHeight`。
- 标题关键词或类名关键词命中其一；若两个关键词列表都为空，则仅靠可见性 + 尺寸规则即可判定。

## 6. 配置最小闭环

核心配置在 `src/GuardService/appsettings.json` 的 `Guard` 段：

- `PollIntervalSeconds`
- `TargetProcessName`
- `TargetExecutablePath`
- `DesktopTitleKeywords`
- `DesktopClassKeywords`
- `MinimumDesktopWidth`
- `MinimumDesktopHeight`
- `RequireVisibleWindow`
- `AllowMinimizedWindow`

其中默认目标进程是：

`Ecloud Cloud Computer Application.exe`

## 7. 两个辅助工具的价值

## 7.1 WindowInspector（结构观察）

用途：当主服务“识别不准”时，先看窗口树事实数据，避免拍脑袋调规则。  
产出：`docs/samples/ecloud-window-elements*.md`

## 7.2 OcrProbe（实验验证）

用途：在固定区域做 OCR，验证某些页面状态能否通过文字可靠区分。  
产出：`docs/samples/ocr/*.md`, `*.json`, `*.png`

它目前是“旁路实验工具”，而不是 GuardService 主路径依赖。

---

## 8. 当前技术取舍（为什么这样设计）

1. **先元数据后视觉**：先用进程/窗口元数据做低成本判定，再考虑 OCR 加强，降低系统复杂度。
2. **先可解释后智能化**：规则可读、日志可追溯，便于在线上快速定位误判来源。
3. **先观测后自动化**：MVP 只做识别，不做恢复动作，避免错误动作放大风险。

这符合“先建立可靠感知，再执行控制”的工程顺序。

---

## 9. 已知风险与约束

- 仅支持 Windows（Win32 API / Windows OCR API）。
- Chromium/Electron 客户端常出现浅 HWND 树，内部 UI 不一定可由子窗口枚举得到。
- 多实例同名进程场景下，路径和启动时间策略可能仍需业务侧加强约束。
- 仅靠窗口元数据对复杂异常态的区分能力有限，后续可能需要 OCR 或更丰富探针。

---

## 10. 建议的下一步迭代（按最小增量）

1. 基于真实样本完善关键词配置（先提升 `DesktopReady` 精度）。
2. 固化误判样本回放流程（先提升规则迭代效率）。
3. 将 OcrProbe 结果纳入“可选二级判定”而非默认主流程。
4. 当识别稳定后，再讨论告警与自动恢复动作。

---

## 11. 一句话结论

这是一个“**云桌面客户端状态识别基座**”项目：当前阶段聚焦于**稳定识别与可解释日志**，通过 `GuardService + WindowInspector + OcrProbe` 形成了从在线判定到离线采样验证的最小闭环。
